# 1차 가공 단계:
- input: raw data
- output: 문서 성격에 따라 추가 가공하기 쉬운 형태로 1차 가공 산출물(.md or .json)

# 2차 가공 단계:
- input: 1차 가공 산출물(.md or .json)
- output: 1차 가공 산출물을 db에 저장, 1차 가공 산출물의 내용을 분류 체계(폴더/파일/스키마)에 맞춰 재조립하여 DB에 저장(2차 가공 산출물)
- manipulate: 2차 가공 산출물이 기존 db에 내용을 추가하는 거니 like git conflict처럼 사람이 조절

# 3차 가공 단계:
- input: 2차 가공 산출물
- output: 2차 가공 산출물을 RAG db에 저장, 2차 가공 산출물을 지식그래프 db에 저장
- manipulate: 기존 db와 충돌 시 충돌 조정 like git conflict

# 문서 검색 단계: 
- 목적에 따라 검색 방법을 다르게 설계
- index
- 이전 검색 cache 기능 
- 답변 근거 출력

# 검색 목적 
1. **조건 기반 가상 테스트**: “Fireball, Enemy, SafeZone, CooldownReady” → *Cast 거부되어야 함*
2. **조합 폭발 테스트**: 사람이 놓치기 쉬운 `타겟=없음 + 타격수=0 + PvP=Off + 광역스킬` 같은 케이스 생성
3. **회귀 테스트**: 룰 바뀐 뒤 “이 변경으로 기존 규칙 127개 중 뭐가 영향받는지”
4. **테스트 공백 탐지**: “전체 조건 축 조합 중 테스트되지 않은 영역 18개 존재”
5. **리스크 기반 추천**: “이 스킬은 PvP/안전지역/아군 타격 축에서 충돌 가능성 높음”
6. **변경 영향 분석**: “화염볼 범위 증가 → 멀티 히트 규칙 재검증 필요”
7. 구현 전에 **기획 결함 발견**: “이건 애매한데?”를 사람 대신 찾아냄
8. 정책 위반 발견: “이 변경으로 인해 Fireball이 ‘광역 고위험 스킬’이라는 의도에서 ‘안전한 범용 스킬’로 변질됨”
9. 밸런스 시뮬레이터 (사람이 못 하는 영역): 특정 룰을 악용하는 전략 탐색
10. 의견 전달: “의도에는 맞지만,초보자 영역에서 오인 사용 확률이 증가합니다.”
11. 게임 화면 이해: 현재 게임이 지금 어떤 상황인지 파악 
12. 게임 조작: 게임 상황에 맞는 다음 조작 제시
